# Viter

Сучасна інфраструктура веб-розробки. Viter - це високорівневий інструмент для автоматизації рутини авторів сучасних бібліотек і застосунків на JS.

## Принципи

- задовольняти потреби сучасних проєктів
- мінімізувати кількість опцій і площину публічного інтерфейсу
- першокласна підтримка TypeScript
- є припустимою жертва швидкістю виконання заради коректності та надійності

## Керівництво

### Проєкт

Код на Viter зберігається в проєктах. Проєкти зберігають усі налаштування та код, необхідний для збірки та публікації бібліотек і застосунків всередині.

Кожен проєкт є каталогом з наступною структурою:

```
<project>/
├── src/
├── viter.js
└── package.json
```

### Пакет

Viter вводить правила, які спрощують организацію коду в межах проєкту, але код бібліотек не має сенсу, якщо бібліотеку ніхто не зможе встановити. Головний інтерфейс npm-екосистеми, пакет, ми знаходимо вкрай невдалим, але мусимо його реалізувати, бо інакше Viter-проєкти не матимуть сенсу для користувачів пакетних менеджерів або CDN.

Для початку роботи з пакетами необхідно встановити поле `version` у кореневому `package.json`. Детальніше про версії пакетів ви зможете прочитати згодом.

Пакети у Viter генеруються з конфігурації у файлі `viter.js`.

```ts
export default viterConfig({
	manifest: name => ({
		name: `@my-scope/${name}`,
	}),
	packages: {
		'first-package': {
			export: [
				// експортовані модулі
			],
			manifest: {
				// специфічні поля `package.json`
			},
		},
	},
})
```

Рядки в масиві `export`, які закінчуються на `-`,  замінюються на всі модулі, ім'я яких починається з цього рядку. Наприклад, `viter-` буде експортувати модулі з назвами `viter-project`, `viter-cache`, `viter-pool`, тощо.

#### Володіння модулями

Кожен модуль з перерахованих в `exports` пакету позначається Viter як "власність" цього пакету. Володіння додає два правила:

- спроба позначити один модуль власністю декількох пакетів призведе до викидання помилки
- при спробі включити в компіляцію пакету `a` модуль, привласнений пакетом `b`, джерело імпорту модулю буде замінене на `b`, а сам пакет `b` буде автоматично додано в список залежностей пакету `a`

Цей механізм гарантує, що опублікований в npm код не буде дублюватись.

#### `license`/`copying`

В дистрибутиви пакетів Viter копіює файл `license` або `copying` з кореню проєкту, якщо такий існує.

#### `readme.md`

В дистрибутив пакету `x` Viter копіює `readme.md` з `src/x.md`, якщо такий існує.

#### `package.json`

Вміст `package.json` згенерованого пакету залежить від багатьох джерел (впорядковані за зростанням пріоритету):

1. наступні поля мають статичні значення за замовчуванням: `type`, `main`, `types` (ніколи не перевизначайте їх)
2. значення наступних полів копіюються з `package.json` у корені проєкту: `version`, `bugs`, `repository`, `homepage`, `funding`, `author`, `contributors`, `license`, `licenses`
3. значення наступних полів автоматично генеруються Viter: `name`, `dependencies`
4. копіюються всі поля з об'єкту, повернутого функцією `manifest` з конфігурації проєкту
5. копіюються всі поля з об'єкту `manifest` конфігурації пакету

#### `version`

Версія кожного пакету береться з версії проєкту. Оскільки Viter дозволяє опубліковувати лише всі пакети разом, можливі "порожні" релізи, які нічого не змінюють в окремому пакеті та його залежностях. Ця поведінка може здаватись дивною, але ми не змогли довести значність проблеми, тому ми ще не реалізували автоматичний пропуск порожніх релізів, хоча розглядаємо таку можливість.

#### `dependencies`

Згенерований список залежностей кожного пакету складається з використаних у коді модулів пакету зовнішніх пакетів і [пакетів-власників використаних модулів](#володіння-модулями). Версії зовнішніх залежностей беруться зі вказаних у `package.json` у корені проєкту.

Наразі Viter не підтримує `peerDependencies` та `optionalDependencies`.

## CLI

- `[]` - опціональний параметр
- `<>` - обов'язковий параметр
- `project` - шлях до каталогу проєкту (за замовчуванням використовується робочий каталог)
- `module` - ім'я модулю
- `app` - ім'я модулю-застосунку
- `script` - шлях до сценарію

### `viter pack`

Компілює всі пакети та запускає тести використаних модулів.

#### `--publish`

Після успішної компіляції пакетів публікує їх в npm.

### `viter run <script>`

Компілює та запускає сценарій. Після `--` можна передати аргументи для CLI Node.js, наприклад, `viter run debug-me.ts -- --inspect 0707`.

### `viter run <module>`

Компілює та запускає тести модулю та всіх його залежностей.
