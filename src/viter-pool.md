# `viter-pool`

![](./viter-pool-illustration.webp)

Ціль цього модуля - забезпечити небойлерплейтну інфраструктуру для описання моделей, які реалізують важку логіку, що її варто виконувати в окремих нитках для ефективного використання паралелізму.

## Використання

Модуль експортує абстрактний клас `ViterPool`, який потребує перевизначення властивості `url`. Вона завжди має повертати релевантне значення `import.meta.url`. Головним інтерфейсом цього класу є `inWorker`. Це проксі, який дозволяє наказувати виконати певний метод моделі її "двійнику" у світі робочих ниток. Розглянемо приклад:

```ts
export class Calculator extends ViterPool {
	url() {
		return import.meta.url
	}

	sum(a: number, b: number) {
		return this.inWorker.sum(a, b)
	}

	sumWork(a: number, b: number) {
		return a + b
	}
}
```

Уявімо, що `sumWork` надовго блокує головне нитку, тому ми створили для нього фасад `sum`, який переносить роботу в окрему нитку. Як це працює?

По-перше, для виконання роботи нам потрібен робітник. Створювати робітника на кожний вилкик і обмежувати його життя цим викликом буде доволі непродуктивно. Рішенням є шаблон [Thread Pool](https://en.wikipedia.org/wiki/Thread_pool), який полягає в перевикористанні обмеженої кількості ниток.

Тепер, коли в нас є робітник, необхідно прояснити йому задачу відправивши повідомлення з наступними відомостями:

- URL модулю (саме для цього потрібно було перевизначати `url()`)
- ім'я класу
- ідентифікатор моделі (про нього - згодом)
- ім'я методу
- серіалізований список аргументів виклику

Потім, виконання процесу призупинюється аж доки робітник не відповість із результатом роботи або помилкою. Як і в Tyger, "помилковість" результату визначається приналежністю до класу `Error`, тому будьте обережні, щоби робітники не кидали нічого зайвого.

Нагадаємо, що серіалізація аргументів і результатів здійснюється NodeJS автоматично через [v8 Serialization API](https://nodejs.org/api/v8.html#serialization-api).

## Створення і налаштування моделей

Як би ми не хотіли в Tyger відмовитись від об'єктів конфігурацій на користь перевизначень функціональних властивостей, але такі об'єкти потрібні для створення моделей-двійників. Кожен нащадок `ViterPool` має бути створений через статичний регістр `fromId`. Значення ключа цього регістру зберігається в моделі в полі `id` і може містити бути будь-яке значення, що його може серіалізувати v8:

```ts
export class FileProcessor extends ViterPool<string> {
	url() {
		return import.meta.url
	}

	process() {
		this.inWorker.heavyProcess()
	}

	file() {
		return TygerFile.absolute(this.id)
	}

	heavyProcess() {
		doSomethingHeavy(this.file())
	}
}

export function fileProcess(file: TygerFile) {
	// НЕПРАВИЛЬНО

	return FileProcessor.make({
		file: () => file,
	}).process()

	// ПРАВИЛЬНО

	return FileProcessor.fromId(file.path).process()
}
```
